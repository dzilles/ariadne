# Ariadne Implementation Plan

This document serves as the step-by-step roadmap for implementing the autonomous V-Model architecture defined in `docs/architecture_plan.md`.

## Phase 1: Core Infrastructure (JIT & Context)
*Goal: Replace the old `activate_ticket` system with an invisible, context-aware Just-In-Time guard.*

- [ ] **Remove Old Guard:** Delete the `activate_ticket` tool and logic from `src/workflows/enforcement.py` and the `RequirementsAgent`.
- [ ] **Build Backend Context:** Implement a thread-safe context manager (or use LangGraph state) to silently store the `active_ticket_id` currently being processed by an agent.
- [ ] **Implement `@jit_vmodel_guard`:** Create a Python decorator for write/commit tools. It should:
    1. Retrieve the `active_ticket_id` from the hidden context.
    2. Query Plane for that ticket's status and blockers.
    3. Block execution and return a clear LLM-friendly string if the ticket is blocked or in the wrong state.

## Phase 2: Orchestrator Autonomy & Delegation
*Goal: Turn the Orchestrator into the "brain" that runs the factory line.*

- [ ] **Build `delegate_to_agent` Tool:** Create a tool specifically for the Orchestrator that allows it to invoke the `chat()` method of other agents programmatically. It must pass the target `ticket_id` into the hidden context.
- [ ] **Standardize Escalations:** Define the strict string formats (`[SUCCESS]`, `[ESCALATION: <reason>]`) that specialized agents must return when their loop finishes.
- [ ] **Update Orchestrator Prompt:** Completely rewrite the Orchestrator's `_get_system_message()` to teach it how to read an Epic's sub-tasks, delegate them sequentially, and handle `[ESCALATION]` responses by re-routing.

## Phase 3: Plane Integration Enhancements
*Goal: Ensure Plane tools support the exact labels and relations required by the architecture.*

- [ ] **Label Management:** Update `PlaneInteraction` and `pm_tools.py` to ensure agents can easily add and remove specific labels (`Review Passed: QA`, `Review Passed: Human`, `Review Failed: Findings`).
- [ ] **Sub-task / Relation Reading:** Ensure tools accurately report `blocked_by` and parent-child Epic hierarchies to the agents.

## Phase 4: Git Workflow Enforcement
*Goal: Implement the isolated branching and merging strategy.*

- [ ] **Expand `GitAgentTools`:** Add tools specifically for the Orchestrator (e.g., `merge_branch`, `abort_merge`).
- [ ] **Conflict Handling:** Ensure the `merge_branch` tool throws a specific catchable exception on conflict, allowing the Orchestrator to halt and alert the human.
- [ ] **Branch Restrictions:** Update the prompts of the Developer, Tester, and Architect to strictly enforce their namespace branches (`feat/`, `test/`, `docs/`) and prevent them from attempting to merge to `main`.

## Phase 5: Agent Prompt Engineering
*Goal: Finalize the instructions for all specialized agents.*

- [ ] **PO Agent:** Teach it to create the 5 standard sub-tasks and link them via `blocked_by` when an Epic is requested.
- [ ] **Requirements & Architect:** Enforce strict markdown formatting, Mermaid.js usage, and reading from the `docs/` folder.
- [ ] **Developer:** Enforce reading specs first, using the playground shell tools for dependencies, and conventional commits.
- [ ] **Tester:** Ensure it explicitly branches off the `epic/` branch, reads Developer code, and executes `pytest`.
- [ ] **QA Agent:** Teach it the Dual-Gate logic (checking for Developer/Tester success, applying Plane Labels, generating markdown reports for systemic failures).
